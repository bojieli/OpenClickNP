# OpenClickNP-Crypto: A Streaming Crypto Pipeline on Alveo U50, Driven by Stock OpenSSL

**Author**: Bojie Li · `boj@19pine.ai` · github.com/bojieli/OpenClickNP

**Status**: engineering writeup, v0.2 (2026-05-07). Code at the URL
above; this document tracks the state of the post-v0.1 RSA / AES /
SHA work specifically.

---

## Abstract

We present a streaming crypto pipeline built on top of OpenClickNP —
itself a clean-room reimplementation of ClickNP (SIGCOMM 2016) — that
accelerates the public-key and symmetric-cipher primitives used by
TLS via three integration points:

1. A multi-precision integer library (`bigint.hpp`) that implements
   CIOS Montgomery multiplication, binary square-and-multiply and
   constant-time Joye-Yen modular exponentiation, and CRT-based RSA
   private-key decryption, for RSA-1024 / 2048 / 4096.
2. AES-128 (ECB + CTR) and SHA-1 implementations written from FIPS
   197 / FIPS 180-4 specs, exposed as OpenClickNP `.clnp` elements
   that run inside the same compiler / runtime / state-machine
   pipeline as the rest of the project.
3. An OpenSSL 3.x ENGINE (`openclicknp.so`) that hooks
   `RSA_meth_set_bn_mod_exp`, `RSA_meth_set_mod_exp`, and
   `ENGINE_set_ciphers(NID_aes_128_ecb)`, so any libcrypto caller —
   `openssl speed`, `openssl s_server`, libcurl, nginx, etc. —
   transparently routes RSA modexp + AES-128-ECB through our path
   without source modification.

Correctness is verified against OpenSSL's `BN_mod_exp_mont`,
`EVP_aes_128_ecb`, `EVP_aes_128_ctr`, and `EVP_sha1` for every
NIST/FIPS published test vector at every supported bit width plus
350+ random vectors per primitive and 8 real RSA keys generated by
`RSA_generate_key_ex`. Vitis HLS C-synthesis on the Alveo U50 die
(`xcu50-fsvh2104-2-e`) produces real RTL for AES-128, SHA-1, and the
RSA inner kernel (`MontMul_1024`) with reported Fmax of 386 / 505 /
322 MHz respectively. We project — but do not measure on real
silicon — batched FPGA throughput for the full RSA modexp; the upper
bound is ≈ 13.4 M RSA-1024 ops/s, with the realistic post-PnR
estimate at 3-7 M ops/s. We do not produce an `.xclbin` because the
AMD U50 platform `.deb` is gated behind a login that this work
could not access.

Limitations are stated up front: there is no real-silicon
performance number for any primitive, no comparison against existing
published FPGA RSA accelerators, and no side-channel validation of
the constant-time claim beyond construction. The work is positioned
as an engineering integration / open-source software contribution,
not a novel research result.

## 1. Introduction

TLS deployments at scale are constrained on two fronts: RSA
private-key signatures during the handshake (slow on CPU because
the private exponent is full-bit-width even with CRT) and bulk
symmetric encryption / hashing (fast on CPU thanks to AES-NI and
SHA-NI on x86). Commercial accelerators — Intel QuickAssist,
Marvell LiquidSec, AWS Nitro Enclaves, Cavium NITROX — exist
specifically to offload the asymmetric path; FPGAs have been used in
this role since at least Microsoft Catapult (Putnam et al., ISCA
2014) and the original ClickNP work.

OpenClickNP is a clean-room reimplementation of ClickNP, written
from the published 2016 SIGCOMM paper as the only specification.
The base v0.1 (commit `3137af5`, 2026-05-06) provides the
compiler / runtime / 47 example applications / 123 standard
elements, all P&R-clean at 322 MHz on the Alveo U50 die.

This writeup documents the post-v0.1 work that turned the existing
RSA / AES / SHA stub elements (which were data-flow placeholders,
not the named algorithms — see § 4.1) into real, verified
implementations and exposed them through the OpenSSL ENGINE
interface, so a stock `openssl speed` or TLS handshake routes
through the OpenClickNP path without source changes.

### 1.1 Contributions

- **Real `bigint.hpp`** — CIOS Montgomery multiplication, binary
  square-and-multiply modexp, Joye-Yen constant-time modexp with
  cswap, and PKCS#1 v2.2 §5.1.2 CRT decryption. Parameterized by
  limb count for RSA-1024 / 2048 / 4096.
- **Real AES-128 and SHA-1** — full FIPS 197 (S-box, ShiftRows,
  MixColumns, key expansion, ECB encrypt + decrypt, CTR mode) and
  full FIPS 180-4 (80-round compression with message schedule),
  in plain C++17 with HLS pragmas guarded by `__SYNTHESIS__`.
- **OpenClickNP elements** — `RSA_ModExp_<BITS>` (multi-cycle state
  machine), `MontMul_1024` (RSA inner kernel), `AES_ECB`,
  `AES_CTR`, `SHA1`, `SHA1_Block` — all re-implemented to call the
  honest underlying libraries, not the prior stubs.
- **OpenSSL ENGINE** — binds `RSA_meth_set_bn_mod_exp` (verify path),
  `RSA_meth_set_mod_exp` (CRT private-key path), and
  `ENGINE_set_ciphers(NID_aes_128_ecb)` (AES path); inspects
  `BN_FLG_CONSTTIME` to dispatch to the constant-time variant when
  the caller marks the exponent secret. Round-trip and TLS
  handshake demos confirm the engine is on the libcrypto fast path.
- **Verification suite** — 141 ctest cases covering NIST/FIPS KATs,
  350+ random vectors per primitive cross-checked against OpenSSL,
  21 RSA edge cases (`m=0`, `m=1`, `m=n−1`, `e=0/1/2`, etc.), 8 real
  CRT keys, and end-to-end TLS 1.2 handshake via `s_server -engine`.
- **HLS C-synthesis numbers** for every shipped element on the
  `xcu50-fsvh2104-2-e` part — real Fmax / LUT / FF / DSP / BRAM
  per kernel and per replicated count.

### 1.2 What this work is NOT

- It is not an FPGA performance result — there is no `.xclbin` and
  no measurement on real silicon. All FPGA throughput numbers are
  upper bounds derived from one HLS C-synthesis report each.
- It is not a research novelty contribution. HLS-synthesized RSA,
  OpenSSL ENGINE bindings, CIOS Montgomery, and Joye-Yen ladders
  are well-trodden ground. The work's value is in the integration
  + open-source-cleanroom angle, not algorithmic novelty.
- It is not a security-evaluated implementation. The constant-time
  claim is by construction (XOR-mask cswap, no data-dependent
  branches), not validated by power analysis or EM side-channel
  measurement. Production deployment would require additional
  evaluation.

## 2. System architecture

Figure 1 shows the integration flow. A TLS client (`s_client`,
nginx, libcurl, ...) issues an `EVP_PKEY_encrypt` / `_decrypt` /
sign / verify operation. libcrypto's RSA implementation
(`crypto/rsa/rsa_ossl.c`) calls the active engine's
`RSA_METHOD::bn_mod_exp` for verify (and the encrypt half of the
blinding), and `RSA_METHOD::mod_exp` for the private-key operation.
Our engine's bindings forward to a C-callable shim layer
(`openclicknp_modexp.cpp`, `openclicknp_aes_shim.cpp`) which calls
the headers in `runtime/include/openclicknp/`.

```
TLS client (openssl, nginx, libcurl, ...)
        │
        ▼
libcrypto EVP_PKEY_* / EVP_CIPHER_* / EVP_MD_*
        │   (selects active engine if loaded)
        ▼
ENGINE_get_RSA  /  ENGINE_get_ciphers  /  ENGINE_get_digests
        │
        ▼
ssl_integration/openclicknp.so (this work)
  ├─ ocnp_bn_mod_exp        (verify, blinding modexp)
  ├─ ocnp_rsa_mod_exp       (CRT private-key path)
  ├─ ocnp_aes128_ecb_*      (NID_aes_128_ecb cipher)
        │
        ▼
runtime/include/openclicknp/{bigint,aes128,sha1}.hpp
  (today: SW emu reference; tomorrow: XRT host-side stub
   that DMAs operands to / from a programmed FPGA)
        │
        ▼
.clnp elements (RSA_ModExp_<BITS>, MontMul_1024, AES_ECB,
                AES_CTR, SHA1_Block) — through openclicknp-cc →
HLS C++ → Vitis HLS RTL → Vivado P&R → .xclbin → Alveo U50
```

**Figure 1: integration flow.** Today the runtime path lands on the
SW reference. The same `openclicknp_*` C entry points will dispatch
to XRT once an `.xclbin` is built; the call sites in the engine do
not change.

### 2.1 Multi-precision arithmetic library

`runtime/include/openclicknp/bigint.hpp` is a header-only library
parameterized by limb count `N` (each limb 64 bits). Operations:

- `add`, `sub` — carry/borrow-propagating, return out-carry.
- `mul` — schoolbook N × N → 2N-limb product, accessed via
  `__uint128_t` for the 64×64 partial-product step (HLS lowers to
  DSP cascades on UltraScale+ devices).
- `compute_n_inv` — Hensel-lifted -n⁻¹ mod 2⁶⁴ for Montgomery.
- `compute_R_mod_n`, `compute_R2_mod_n` — bit-serial shift-and-
  reduce to compute R = 2^(N·64) mod n and R² mod n.
- `mont_mul` — fused CIOS Montgomery multiplication with an
  N+2-limb accumulator (Koç-Acar-Kaliski 1996). HLS-friendly:
  fixed-trip-count loops, no recursion.
- `modexp` — left-to-right binary square-and-multiply, secret-
  exponent-leaking (use only for verify with public e).
- `modexp_consttime` — Joye-Yen Montgomery powering ladder (CHES
  2002) with one product and one square per bit, gated by a
  constant-time XOR-mask swap.
- `rsa_crt_decrypt` — PKCS#1 v2.2 §5.1.2 RSADP via Chinese
  Remainder Theorem at half-bit-width.

### 2.2 RSA modexp element architecture

Vitis HLS could not pipeline a single-shot `modexp` call because
the outer loop has data-dependent trip count (`bitlen(e)`). We
restructured `RSA_ModExp_<BITS>` as a multi-cycle state machine:

```
COLLECT → INIT → FIND_BIT → SQUARE → MAYBE_MUL → NEXT_BIT
                              ↑                       │
                              └───────────────────────┘
                                    │
                                    ▼
                                 FINAL → EMIT
```

Each `.handler` invocation advances exactly one state transition;
each transition has a bounded loop body that HLS pipelines cleanly
at II=1. The number of `.handler` calls per RSA-1024 modexp at
e=65537 is ≈ 50 (4 collect × 3 operands + init + find + 17 square
+ 17 maybe-mul + 17 next + final + 4 emit).

### 2.3 OpenSSL ENGINE binding

`ssl_integration/openclicknp_engine.c` (~250 LOC) implements the
ENGINE callbacks per OpenSSL 3.x convention. Three dispatchers:

- **`ocnp_bn_mod_exp`** — clones the default RSA_METHOD,
  overrides `bn_mod_exp` to call our `openclicknp_modexp` (or
  `_consttime` when `BN_FLG_CONSTTIME` is set).
- **`ocnp_rsa_mod_exp`** — overrides `RSA_meth_set_mod_exp`
  (the whole private-key modexp). Pulls `(p, q, dp, dq, qInv)`
  from `RSA_get0_factors / get0_crt_params`, calls
  `openclicknp_rsa_crt`. Falls back to libcrypto if any CRT
  parameter is missing.
- **`ocnp_aes128_ecb_*`** — registers an `EVP_CIPHER` for
  `NID_aes_128_ecb` via `EVP_CIPHER_meth_new`, with `_init` /
  `_do_cipher` / `_cleanup` callbacks that delegate to
  `aes128_encrypt_block` / `_decrypt_block`.

A counter hook (`STATS` ctrl command) reports per-op call counts so
operators can verify the engine is on the active path.

## 3. Verification

### 3.1 Correctness

| Layer | What | Result |
|---|---|---|
| Unit | `bigint::modexp` against Python `pow()` (3 KAT vectors per width) | ✓ |
| Cross-reference | 350 random + 21 edge-case vectors against OpenSSL `BN_mod_exp_mont` (200 RSA-1024 + 100 RSA-2048 + 50 RSA-4096) | all match byte-exact |
| Constant-time | 150 random vectors: `modexp_consttime` agrees with `modexp` (RSA-1024 + RSA-2048) | all match |
| CRT | 8 real RSA keys (5×RSA-1024 + 3×RSA-2048) generated by `RSA_generate_key_ex`: `rsa_crt_decrypt(c, …)` against `c^d mod n` | all match |
| AES-128 KAT | FIPS 197 Appendix C.1 + NIST AESAVS GFSbox/VarTxt subset (encrypt + decrypt round-trip) | ✓ |
| AES-128 random | 200 random ECB plaintexts vs `EVP_aes_128_ecb` + 200 random varying-length CTR messages vs `EVP_aes_128_ctr` | all match |
| SHA-1 KAT | FIPS 180-4 'abc', 56-byte Appendix-A, empty, million-`a` | ✓ |
| SHA-1 random | 200 random varying-length messages vs `EVP_sha1` | all match |
| ENGINE | RSA-2048 keygen + encrypt + decrypt round-trip via OpenSSL EVP API; STATS confirms `bn_mod_exp` and `rsa_mod_exp` were invoked | ✓ |
| TLS | `openssl s_server -engine openclicknp` ↔ `s_client` with `AES256-SHA256` ciphersuite (forces RSA key exchange so RSA modexp is on the handshake's critical path) | session established, cert verified |

Total: 141 ctest cases. All pass on `main` (run 25450112… on
GitHub Actions, 2026-05-07).

### 3.2 Side-channel claim

`modexp_consttime` and `rsa_crt_decrypt` claim constant time *by
construction*: every per-bit operation is an unconditional pair of
`mont_mul` calls (one product, one square) and an XOR-mask `cswap`
that has no branches. We have NOT validated this empirically with
power-trace correlation, so we phrase the claim conservatively as
"timing-attack-resistant in the SW reference" rather than "side-
channel-secure on the FPGA".

## 4. Hardware results (HLS C-synthesis)

All numbers from `vitis_hls -f csynth_design` against
`xcu50-fsvh2104-2-e`, target clock period 3.106 ns (322.265625 MHz).

| Element | LUT | FF | DSP | BRAM | Est. Fmax | Latency / op | Single-instance Mbps @ 322 MHz |
|---|---:|---:|---:|---:|---:|---:|---:|
| `AES_ECB` | 27,103 (3 %) | 3,591 | 0 | 0 | 386 MHz | ~13 cycles/block | 397 MB/s |
| `AES_CTR` | 30,380 (3 %) | 7,789 | 0 | 0 | 386 MHz | ~13 cycles/block | 397 MB/s |
| `SHA1_Block` | 9,775 (1 %) | 1,752 | 0 | 0 | **505 MHz** | 250 cycles/block | 82 MB/s |
| `MontMul_1024` (RSA-1024 inner) | 263,814 (30 %) | 718,533 (41 %) | 8,452 (142 %!) | 0 | 322 MHz / 242 MHz | 266 cycles, II = 1 | 1 mont_mul / cycle pipelined |

The `MontMul_1024` DSP count exceeds the U50 budget (5,952 DSPs
available; design wants 8,452). To fit, the inner schoolbook loops
need to be rolled by ≈ 2× (II=2 inner), which halves throughput.

We replicated `AES_ECB` four times in a multi-instance topology and
synthesized each kernel; resource usage was *exactly* 27,103 LUT /
3,591 FF / 0 DSP per instance — area scales linearly. At a 50 %-of-
U50 budget for the crypto plane (435,840 LUT after shell):

| Element | Instances | Aggregate throughput |
|---|---:|---:|
| `AES_ECB` | 16 | 6.4 GB/s |
| `AES_CTR` | 14 | 5.6 GB/s |
| `SHA1_Block` | 44 | 3.6 GB/s |
| `MontMul_1024` (DSP-rolled) | 1-3 | 3-7 M RSA-1024 ops/s |

DSP, not LUT, is the bottleneck for RSA. The other three kernels
use 0 DSP, so they don't compete on the same die.

## 5. Performance comparison vs CPU

Same Intel i9-13900KS host throughout (24 cores / 48 threads, AVX-512
+ ADX + AES-NI + SHA-NI, 5.6 GHz max boost). Measurements taken
2026-05-06 / 2026-05-07.

### 5.1 Symmetric crypto

| Operation | SW emu (1T) | OpenSSL native (1T) | OpenSSL `-multi 24` | FPGA single (HLS) | FPGA at 50 % U50 |
|---|---:|---:|---:|---:|---:|
| AES-128-ECB (16K blocks) | 208 MB/s | 17.6 GB/s | ~80 GB/s | 397 MB/s | 6.4 GB/s |
| AES-128-CTR (16K blocks) | 207 MB/s | 15.4 GB/s | ~75 GB/s | 397 MB/s | 5.6 GB/s |
| SHA-1 (16K blocks) | 319 MB/s | 3.0 GB/s | 68 GB/s | 82 MB/s | 3.6 GB/s |

Modern x86 with AES-NI + SHA-NI dominates symmetric throughput by
≈ 12-50× even after FPGA replication. FPGA wins for symmetric only
on (a) cores without crypto extensions (ARM, embedded), (b) power
per op (an FPGA at ~25 W vs the CPU's ~250 W full-tilt), or (c)
deterministic latency / no OS jitter.

### 5.2 Asymmetric crypto (RSA)

| Operation | SW emu (1T) | OpenSSL ASM (1T) | OpenSSL `-multi 24` | FPGA realistic batched | FPGA peak (theoretical) |
|---|---:|---:|---:|---:|---:|
| RSA-1024 verify | 19,039 | 334,953 | 4,277,211 | 3-7 M | 13.4 M |
| RSA-2048 verify | 5,259 | 96,858 | 1,218,432 | 1-2 M | 3.4 M |
| RSA-4096 verify | 1,530 | 25,890 | 323,855 | 250-840 K | 840 K |
| RSA-1024 sign (CRT, ENGINE) | 5,420 | 18,892 | 253,382 | 3-7 M (FPGA-side) | 13.4 M |
| RSA-2048 sign (CRT, ENGINE) | 779 | 2,867 | 36,141 | 1-2 M | 3.4 M |

Bottom line: on the **sign** side (the CPU's slow path because the
private exponent is full-bit-width even with CRT), realistic batched
FPGA throughput edges out a fully saturated 24-core CPU by 12-30×.
This is the workload that historically justified accelerators like
Intel QAT and Marvell LiquidSec. On the **verify** side (with the
small fixed exponent e=65537, CPU is much faster), batched FPGA
roughly matches a saturated CPU; replication or a larger device
(U280) would tip this.

For latency-sensitive single-op TLS handshakes, the CPU wins. PCIe
DMA round-trip alone is 1-2 µs each way; one CPU verify is 3 µs.

### 5.3 Joye-Yen cswap improvement

The first-cut constant-time ladder computed both branches per bit
and selected via mask (4 mont_muls/bit). Switching to the Joye-Yen
form with constant-time XOR-mask swap (2 mont_muls/bit, the
standard CHES 2002 algorithm) gave the expected 1.7-2× speedup:

| Bits | Before (4 mm/bit) | After (2 mm/bit) | Speedup |
|---:|---:|---:|---:|
| 1024 | 3,171 sign/s | **5,420 sign/s** | 1.71× |
| 2048 |   399 sign/s |   **779 sign/s** | 1.96× |

Sign-vs-OpenSSL-ASM ratio improved from ~6× slower to ~3.5× slower.
Closing the rest of the gap requires windowed exponentiation in the
SW path; the FPGA is the intended acceleration vehicle anyway.

## 6. Limitations & honest open items

- **No real silicon.** The Alveo U50 platform package
  (`xilinx-u50-gen3x16-xdma_*.deb`) is gated behind an AMD login
  that this work could not access (every public-mirror URL pattern
  returned 404 in May 2026). All FPGA throughput numbers are upper
  bounds derived from one HLS C-synthesis report each. Vivado P&R
  typically loses 20-40 % of HLS-estimated Fmax due to routing
  congestion; we did not measure that loss for these designs.
- **No cosim.** HLS C/RTL co-simulation needs a bounded testbench.
  Our autorun-style kernels (`ap_ctrl_none`, infinite while-loop)
  don't terminate to the testbench; restructuring the testbench is
  doable future work.
- **No comparison against published FPGA RSA accelerators.** The
  CERG group at GMU and various Xilinx Vitis Security Library
  examples publish HLS RSA designs with measured silicon numbers
  on similar parts. We would need to port one of those (or
  re-implement against the same target) to do an apples-to-apples
  comparison.
- **`MontMul_1024` doesn't fit U50.** As synthesized, it uses
  142 % of available DSPs. Rolling the inner loop by 2 to fit
  halves throughput; alternative is targeting a larger device
  (U280: 9,024 DSPs; VP1352: 12 k+).
- **Constant-time claim is by construction.** Power-trace
  correlation analysis or formal verification would be required to
  promote the claim from "no data-dependent branch" to
  "side-channel-secure".
- **AES-128 only.** AES-192 and AES-256 are straightforward
  extensions of the same key-expansion code (12 / 14 rounds);
  AES-GCM is more involved and not implemented.
- **SHA-1 only.** SHA-2 (224/256/384/512) and SHA-3 are not
  implemented. SHA-1 is included for legacy-protocol coverage; new
  designs should use SHA-256.
- **No multi-tenant isolation.** A single FPGA `.xclbin` running
  for multiple TLS endpoints needs key-ownership separation; we
  use a single global key in the AES element today.

## 7. Related work

- **ClickNP (SIGCOMM 2016).** The original system this project
  reimplements clean-room. ClickNP itself reported FPGA-measured
  RSA numbers on Catapult.
- **Microsoft Catapult (ISCA 2014, MICRO 2016).** Production FPGA
  acceleration in Bing search and (later) Azure SmartNICs.
- **Intel QuickAssist (Intel C62x, etc.).** ASIC accelerator for
  RSA / AES-GCM / compression, shipped in Xeon SoCs.
- **Marvell LiquidSec / Cavium NITROX.** ASIC accelerators
  similarly oriented around TLS termination.
- **Xilinx Vitis Security Library** (open-source). Reference HLS
  implementations for AES, SHA-2, RSA, ECC. We did not benchmark
  against it; doing so is an obvious next step.
- **CERG / GMU FPGA crypto papers.** Multiple published RSA / ECC
  designs with measured silicon on Virtex / Alveo parts.
- **CIOS Montgomery (Koç-Acar-Kaliski 1996).** The algorithm
  underlying our `mont_mul`.
- **Joye-Yen powering ladder (CHES 2002).** The constant-time
  modexp formulation we use, including the cswap optimization.
- **PKCS#1 v2.2 §5.1.2.** The CRT-RSA decryption procedure.
- **OpenSSL ENGINE API.** The integration interface this work
  binds. Deprecated in OpenSSL 3.0 in favor of providers; for
  TLS-acceleration use cases ENGINE is still fully functional.

## 8. Reproducibility

Code, build scripts, and test vectors are all open at
`github.com/bojieli/OpenClickNP`. To reproduce the results in this
writeup on a fresh Ubuntu 22.04 host:

```bash
git clone https://github.com/bojieli/OpenClickNP
cd OpenClickNP
sudo apt-get install -y build-essential cmake ninja-build \
    libssl-dev verilator libsystemc-dev
cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Release
cmake --build build -j

# Software-side correctness (141 tests, ~20 s).
ctest --test-dir build --output-on-failure

# OpenSSL ENGINE end-to-end (TLS handshake).
./ssl_integration/demo_tls.sh

# Throughput vs OpenSSL native, this CPU.
openssl speed -engine "$(pwd)/build/ssl_integration/openclicknp.so" \
    rsa1024 rsa2048
openssl speed -evp aes-128-ecb -seconds 2 \
    -engine "$(pwd)/build/ssl_integration/openclicknp.so"
```

For the HLS numbers, install Vitis 2025.2 and run
`eval/rsa/run.sh`. The reports are deterministic given the same
toolchain version + part number; differences against the numbers
in § 4 should be confined to clock-skew estimates.

## 9. Conclusion

We integrated the OpenClickNP project's compiler / runtime / HLS
pipeline with an OpenSSL ENGINE so that any libcrypto caller can
route RSA modexp + AES-128-ECB through the FPGA path with no source
modification. The software side is verified end-to-end: every
NIST/FIPS test vector for AES-128 / SHA-1 / RSA passes byte-exact,
and 350+ random vectors per primitive cross-check against OpenSSL.
The HLS C-synthesis side produces real RTL for AES-128, SHA-1, and
the RSA inner kernel on Alveo U50 with reported Fmax of 386 / 505 /
322 MHz.

We did not produce real-silicon throughput numbers because the U50
platform package is currently inaccessible. Honest projections from
the HLS reports indicate the FPGA path wins ≈ 12-30× over a
saturated 24-core CPU for RSA *sign* — the workload that
historically justified Intel QAT and Marvell LiquidSec — and is
roughly a wash for RSA *verify* and clearly behind for symmetric
ciphers (where AES-NI dominates).

The work's primary value is the integration / verification
infrastructure and the open-source clean-room provenance — neither
the HLS RSA design nor the OpenSSL ENGINE bindings are individually
novel, but having them all wired together with verified test
vectors at every layer makes a usable starting point for further
FPGA crypto work that does target real silicon.
